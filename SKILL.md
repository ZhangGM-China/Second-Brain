# Knowledge Base Skill

管理个人知识库的三种操作。当用户说"摄入"、"查询知识库"、"检查知识库"或触发此技能时，响应对应操作。

## 核心原则

> 本文件规定 LLM 如何维护个人知识库 Wiki。LLM 必须严格遵循。
> 源文档统一存放在 Obsidian Vault（`~/Documents/Obsidian Vault/`），作为知识库的 raw 层。
> 用户会通过 Claudian 触发 Ingest 流程处理 Vault 中的新文章。

---

## 知识库目录结构

```
knowledge/
├── wiki/                  # LLM 生成的知识库（Git 版本控制）
│   ├── index.md          # 总索引
│   ├── log/             # 每日日志文件夹
│   │   └── yyyy-mm-dd.md # 每日日志
│   ├── 概念/            # 概念页面
│   ├── 实体/            # 实体（人物、项目、工具）
│   ├── 主题/            # 主题页面
│   └── 源文档摘要/       # 每篇源文档的摘要页
└── schema.md            # 本文件（Schema 定义）
```

## Wiki 页面

### 命名规范

- 使用 `-` 分隔英文单词
- 禁止：空格、特殊字符混用
- 中文术语直接引用，英文术语尽量附带中文解释
- 示例：
```
| 文件名 | 说明 |
|--------|------|
| `RAG（检索增强生成）.md` | 英文+中文括号 |
| 幻觉.md` | 中文 |
| machine-learning（机器学习）.md` | 英文单词用‘-’ 分割 + 中文括号 |
```

### Frontmatter 规范

每个 Wiki 页面必须包含以下 frontmatter，严格遵循标准 YAML 格式：

```yaml
---
title: 页面标题
created: "[[YYYY-MM-DD]]"
updated: "[[YYYY-MM-DD]]"
tags:
  - 标签1
  - 标签2
related:
  - "[[页面1]]"
  - "[[页面2]]"
source: "[[源文档路径]]"
---
```

#### 字段说明

- `title`：人类可读标题，用于显示
- `created`：创建日期，创建文档后固定，更新文档时不要更新此日期，格式 `[[YYYY-MM-DD]]`
- `updated`：最后更新日期，每次更新文档时更新，格式 `[[YYYY-MM-DD]]`
- `tags`：标签列表，**每行一个，用 `-` 前缀**（可用多级如 `ai/安全`）
- `related`：相关页面链接，每行一个，`[[页面名]]` 格式
- `source`：如果是摘要/笔记页，标注来源（使用 `[[文件路径]]` 格式，如 `[[001 通往AGI之路/文章名.md]]`，Obsidian 可点击跳转）

## log（日志）

### 日志说明

1. 每次重要操作后，在对应日期的日志文件中记录：
	- 操作类型（ingest/query/lint）
	- 处理内容
	- 关键决策或洞察
2. 保持日志简洁

### 更新规范

- 文件名格式：`yyyy-mm-dd.md`（如 `2026-04-11.md`）
- 放在 `log/` 子文件夹中
- **不需要 frontmatter**
- 格式：
```markdown
## 新增页面
- [[概念/页面名]]：本次操作的简短描述说明
- [[主题/页面名]]：本次操作的简短描述说明

## 更新页面
- [[概念/页面名]]：本次操作的简短描述说明
- [[主题/页面名]]：本次操作的简短描述说明
  
## 健康修复
- lint | 发现并修复的问题
  
## 其他操作
- lint | 对知识库进行的其他操作
```

## 操作一：Ingest（摄入新文档）

当用户提供文章或要求处理文档时：

### 处理流程

1. 定位源文档：
   - 直接读取 `~/Documents/Obsidian Vault` 中的 markdown 文件，忽略文件中的外部链接，只解析本地已有内容
   - 理解核心内容、关键观点、数据、结论

2. 写入源文档摘要 → `~/Documents/Obsidian Vault/knowledge/wiki/源文档摘要/`
	- 文件名：与 Obsidian 文件名一致
   - 注明该源文档涉及到的概念、实体、主题，并将其与对应的页面关联上
	- 内容：核心摘要、关键摘录、个人解读

3. 写入概念页 → `~/Documents/Obsidian Vault/knowledge/wiki/概念/`
	- 源文档中涉及到的所有概念单独建页
   - 将该概念页中提及的相关概念、实体、主题，与其对应的页面关联上
	- 内容：概念定义、关键要点、相关概念

4. 写入实体页 → `~/Documents/Obsidian Vault/knowledge/wiki/实体/`
	- 文章中涉及到的所有人物、组织、项目等物理实体单独建页
   - 将该实体页中提及的相关概念、实体、主题，与其对应的页面关联上

5. 写入主题页 → `~/Documents/Obsidian Vault/knowledge/wiki/主题/`
	- 若文章中存在由多个概念聚合所组成的主题，为该主题单独建页
   - 将该主题页中提及的相关概念、实体、主题，与其对应的页面关联上

6. 更新 `wiki/index.md`
	- 在对应分类下添加条目
	- 格式：`- [[页面名]] — 一句话描述`

7. 更新 `wiki/log/YYYY-MM-DD.md`
	- 按log更新规范的要求更新

### 其他要求

1. 页面关联遵循obsidian语法
   - 词汇与页面的关联需遵循obsidian的语法
   - 例如；[[渐进式披露]]

2. 进行创建或编辑的决策：

| 情况       | 操作               |
| -------- | ---------------- |
| 新主题/新实体  | 新建页面             |
| 现有主题的新洞察 | 添加到现有页面          |
| 现有内容的更正  | 编辑现有页面，标注更新      |
| 新旧冲突     | 编辑旧页面，标注矛盾，写入新页面 |

## 操作二：Query（查询）

当用户询问知识库时：

### 处理流程

1. 先读 `wiki/index.md`查询相关页面
2. 读相关页面获取细节
3. 综合回答，引用来源
4. 重要洞察可归档：如果答案有价值，问用户是否要存为 Wiki 页面

---

## 操作三：Lint（健康检查）

当用户要求检查知识库健康度时：

### 处理流程

- 矛盾检测：同一主题的不同页面是否冲突
- 过时标注：标注需要更新的内容
- 孤儿页面：没有 inbound 链接的页面
- 缺失关联：提到但没有自己页面的概念
- 空洞页面：内容过少需要扩充的概念
- 页面关联：有相关页面却没有建立 inbound 链接的词汇，将词汇链接到对应页面
- 重复概念：两个概念的核心一致，将两个概念合并

### 报告格式
```
## Lint Report [YYYY-MM-DD]

### Issues
- ...

### Suggestions
- ...
```

## 主动性（LLM 多主动）

在进行所有操作时，均鼓励以下行为：
- 主动发现新连接（"这个概念和 XX 有关，要不要关联？"）
- 主动建议 lint（"这部分可能过时了，要不要更新？"）
- 主动归档（"这个问答很有价值，要不要存为页面？"）
- 主动发现空白（"你还没覆盖 XX 领域，要不要探索？"）
